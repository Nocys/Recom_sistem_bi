# BAGIAN 16 — FIX PERMANENT DELETE PRODUCT DENGAN RELATED DATA CLEANUP

## Instruksi Utama untuk Codex

Kerjakan hanya **Bagian 16 — Fix Permanent Delete Product dengan Related Data Cleanup**.

Masalah saat ini:

```text
Saat admin melakukan delete permanen produk, sistem menolak jika produk memiliki aktivitas user.
Pesan error:
"Produk tidak dapat dihapus permanen karena memiliki aktivitas user."
```

Aturan tersebut harus dihapus.

Mulai Bagian 16, admin harus bisa menghapus produk secara permanen walaupun produk tersebut memiliki aktivitas user.

Namun, agar data tidak rusak, backend harus menghapus data terkait produk terlebih dahulu secara aman.

---

# 1. Tujuan Bagian 16

Setelah Bagian 16 selesai:

1. Admin bisa melakukan delete permanen produk.
2. Produk tetap bisa dihapus walaupun memiliki:

   * like
   * wishlist
   * interaction
   * tanya admin
   * page view
3. Backend tidak boleh lagi menolak delete permanen hanya karena produk punya aktivitas user.
4. Data terkait produk harus ikut dihapus.
5. Proses delete permanen harus memakai database transaction.
6. Jika salah satu proses gagal, seluruh proses harus rollback.
7. Frontend admin menampilkan konfirmasi yang jelas.
8. Toast sukses/gagal harus sesuai.
9. Setelah produk dihapus, produk tidak muncul lagi di:

   * daftar produk
   * admin table
   * profile liked products
   * profile wishlist
   * rekomendasi
   * product detail
10. Lakukan pengecekan dan pengujian setelah implementasi.

---

# 2. Aturan Baru Permanent Delete

Aturan lama:

```text
Jika produk punya aktivitas user, permanent delete ditolak.
```

Aturan baru:

```text
Produk tetap boleh dihapus permanen walaupun punya aktivitas user.
Semua data terkait produk ikut dihapus.
```

Jadi endpoint:

```http
DELETE /admin/products/:id/permanent
```

tidak boleh lagi return `409 Conflict` hanya karena produk memiliki aktivitas user.

---

# 3. Data Terkait yang Harus Dihapus

Saat produk dihapus permanen, hapus data dari tabel berikut:

```text
product_likes
product_wishlists
interactions
products
```

Urutan aman:

```text
1. Hapus product_likes berdasarkan product_id
2. Hapus product_wishlists berdasarkan product_id
3. Hapus interactions berdasarkan product_id
4. Hapus products berdasarkan id
```

Gunakan transaction:

```sql
BEGIN;
DELETE FROM product_likes WHERE product_id = $1;
DELETE FROM product_wishlists WHERE product_id = $1;
DELETE FROM interactions WHERE product_id = $1;
DELETE FROM products WHERE id = $1 RETURNING *;
COMMIT;
```

Jika error:

```sql
ROLLBACK;
```

Catatan:

Jika database sudah memakai `ON DELETE CASCADE`, tetap lebih baik service menghapus data terkait secara eksplisit agar response bisa menampilkan jumlah data yang ikut terhapus.

# 5. Backend Service: Permanent Delete

Perbarui:

```text
backend/services/productService.js
```

Cari function permanent delete yang sebelumnya menolak produk dengan related activity.

Kemungkinan nama function:

```js
permanentlyDeleteProduct(productId)
```

atau nama sejenis.

Ubah logic-nya agar:

1. Cek produk ada atau tidak.
2. Jika produk tidak ada, return not found.
3. Hitung jumlah data terkait:

   * likes
   * wishlists
   * interactions
4. Mulai transaction.
5. Hapus data terkait.
6. Hapus produk.
7. Commit transaction.
8. Return deleted product dan jumlah data terkait yang ikut dihapus.
9. Jika error, rollback.

---

## 5.1 Contoh Logic Service

Sesuaikan dengan struktur database connection project.

Jika project memakai `pool`, gunakan client transaction:

```js
async function permanentlyDeleteProduct(productId) {
  const client = await pool.connect();

  try {
    await client.query('BEGIN');

    const productResult = await client.query(
      'SELECT * FROM products WHERE id = $1',
      [productId]
    );

    if (productResult.rows.length === 0) {
      await client.query('ROLLBACK');
      return null;
    }

    const product = productResult.rows[0];

    const likeDeleteResult = await client.query(
      'DELETE FROM product_likes WHERE product_id = $1',
      [productId]
    );

    const wishlistDeleteResult = await client.query(
      'DELETE FROM product_wishlists WHERE product_id = $1',
      [productId]
    );

    const interactionDeleteResult = await client.query(
      'DELETE FROM interactions WHERE product_id = $1',
      [productId]
    );

    const deleteProductResult = await client.query(
      'DELETE FROM products WHERE id = $1 RETURNING *',
      [productId]
    );

    await client.query('COMMIT');

    return {
      deleted_product: deleteProductResult.rows[0] || product,
      deleted_related_activity: {
        likes: likeDeleteResult.rowCount,
        wishlists: wishlistDeleteResult.rowCount,
        interactions: interactionDeleteResult.rowCount
      }
    };
  } catch (error) {
    await client.query('ROLLBACK');
    throw error;
  } finally {
    client.release();
  }
}
```

Sesuaikan nama import `pool` dengan project.

Jangan memakai string interpolation SQL.

Gunakan parameterized query.

# 6. Hapus Rule Lama yang Menolak Delete

Cari logic lama seperti:

```js
if (activityCounts.has_related_activity) {
  throw ...
}
```

atau:

```js
return conflict(...)
```

atau:

```js
status 409
```

Hapus atau ubah logic tersebut.

Permanent delete tidak boleh lagi gagal hanya karena:

```text
interaction_count > 0
like_count > 0
wishlist_count > 0
```

Namun function hitung activity boleh tetap dipakai untuk memberi informasi jumlah data yang ikut dihapus.

# 7. Backend Controller

Perbarui:

```text
backend/controllers/productController.js
```

Pastikan controller permanent delete:

1. Memanggil service permanent delete baru.
2. Jika produk tidak ditemukan, return 404.
3. Jika berhasil, return 200.
4. Jangan return 409 karena related activity.
5. Response sukses menyertakan jumlah data terkait yang dihapus.

Response sukses:

```json
{
  "success": true,
  "message": "Product permanently deleted successfully",
  "data": {
    "deleted_product": {
      "id": "..."
    },
    "deleted_related_activity": {
      "likes": 2,
      "wishlists": 1,
      "interactions": 5
    }
  }
}
```

Response jika produk tidak ditemukan:

```json
{
  "success": false,
  "message": "Product not found"
}
```


```text

```

# 13. Profile Setelah Delete

Karena data:

```text
product_likes
product_wishlists
interactions
```

ikut dihapus, produk yang sudah di-delete permanen tidak boleh muncul lagi di:

```text

```

Jika profile masih menampilkan produk yang sudah delete, berarti query masih mengambil data yang tidak ikut terhapus atau cache frontend belum reload.

````
# 16. Dokumentasi Delete vs Deactivate

Perbarui:

```text
README.md
backend/README.md
frontend/README.md
docs/thesis-notes.md
````

Gunakan penjelasan baru:

```md
## Delete vs Deactivate

Sistem menyediakan dua jenis penghapusan produk:

### Deactivate

Deactivate hanya mengubah status produk menjadi inactive. Data produk dan aktivitas user tetap tersimpan.

Gunakan deactivate jika admin ingin menyembunyikan produk tanpa menghapus histori data.

### Delete Permanen

Delete permanen menghapus produk dari database. Semua aktivitas user terkait produk tersebut, seperti like, wishlist, dan interaction, ikut dihapus.

Gunakan delete permanen jika admin benar-benar ingin menghapus produk dari sistem.
```
