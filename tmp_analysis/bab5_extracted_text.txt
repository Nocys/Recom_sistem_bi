[0005] (Normal) Menentukan kebutuhan sistem menjadi tahap awal pengembangan perangkat lunak dengan metode waterfall. Tahap analisis kebutuhan tidak membahas kode program yang ditulis dan berfokus pada perancangan API chatbot berbasis dokumen PDF dengan kemampuan Retrieval-Augmented Generation (RAG) dan fitur deteksi ujaran kebencian (hate speech detection) menggunakan hybrid BERT.
[0006] (Normal) Kebutuhan sistem yang dibangun didasarkan pada kompleksitas NLP yang melibatkan pemahaman dokumen eksternal dan klasifikasi sentimen negatif. Kebutuhan sistem yang dirancang dikelompokkan menjadi kebutuhan fungsional dan kebutuhan nonfungsional.
[0007] (Normal) Kebutuhan fungsional dalam sistem ditujukan untuk memberikan layanan spesifik sebagai dasar yang harus disediakan oleh sistem, menanggapi input yang diberikan sistem, hingga perilaku sistem dalam situasi tertentu. Adapun fitur-fitur fungsional dalam sistem yang dirancang sebagai berikut:
[0008] (Normal) Kemampuan menerima dan mengekstraksi dokumen PDF berupa kemampuan

[0011] (Normal) dalam menerima input file dokumen dengan format .pdf. Dokumen yang telah diterima kemudian akan dilakukan proses ekstraksi untuk mengambil teks mentah.
[0012] (Normal) Proses pemrosesan teks yang telah diekstrak tersebut kemudian akan dibuah menjadi vektor yang dalam prosesnya melibatkan proses embedding yang menggunakan arsitektur transformer untuk mengubah potongan teks (chunks) menjadi vektor berdimensi tinggi, kemudian hasil embedding akan disimpan dalam database vektor.
[0013] (Normal) Deteksi ujaran kebencian menggunakan model hybrid yang membuat sistem mampu memvalidasi etika bahasa dari pertanyaan input user. Proses ini melibatkan klasifikasi untuk pertanyaan yang menggunakan pendekatan hybrid dengan BERT sebagai extractor (mengambil embedding kontekstual) dan ditambahkan algoritma KNN atau Logistic Regression untuk menentukan apakah teks mengandung hate speech atau netral.

[0015] (Normal) Pencarian semantik (Retrieval) merupakan penencarian potongan teks (chunk) dalam vektor yang memiliki kemiripan tertinggi dengan pertanyaan pengguna.

[0017] (Normal) Generasi jawaban (Generation) untuk mengirimkan prompt ke dalam model
[0018] (Normal) LLM, yaitu SeaLLLM, untuk menghasilkan jawaban yang sesuai dengan pertanyaan dalam bahasa Indonesia atau bahasa Inggris.

[0035] (Heading 1) Gambar 5.1 Arsitektur Sistem Web Penyedia Chatbot

Tugas tersebut meliputi penyediaan antarmuka pengguna untuk melakukan input data dokumen PDF serta interaksi percakapan dengan chatbot. Saat pengguna mengunggah PDF, backend akan melakukan berbagai rangkaian proses yang kemudian menghasilkan hasil akhir berupa penyimpanan informasi dokumen ke dalam database vektor dan menghasilkan API yang dapat digunakan oleh user. [0178] (Heading 2) 5.1.2.5.1 Relational Database Management System
[0179] (Normal) Relational Database Management System (RDMS) atau basis data relasional merupakan basis data terstruktur yang tersimpan dalam format tabel (baris dan kolom) dengan menggunakan bahasa standar SQL. SQL digunakan untuk menyimpan entitas data dengan relasi antartabel. Gambar 5.28 mengilustrasikan basis data relasional.
[0181] (Heading 1) Gambar 5.28 Rancangan Database Relasional
[0182] (Normal) Basis data relasional yang dibangun pada Gambar 5.28 mengilustrasikan hubungan antartabel. 3 tabel utama yang mengatur proses operasional berupa tbuser, tbapi, dan tbchat. Tbuser berfungsi untuk melakukan manajemen atas data pengguna seperti proses membuat akun, login, menentukan data superuser, dan berbagai informasi lainnya. Tbapi akan mengatur seluruh proses yang dilakukan oleh pengguna dalam hal menyimpan dokumen untuk digunakan dalam chatbot dan menghasilkan kode API yang dapat digunakan oleh pengguna. Tabel tersebut mengambil id pengguna menjadi bentuk user_id agar sistem mengenali kepemilikan file tersebut. Tbchat memberikan fungsi di mana pengguna tanpa akun dapat melaporkan keluhan ataupun memberikan masukan atas fungsi tambahan pada sistem, sehingga tbchat meliputi kolom penting seperti Gmail, nama, nomor telepon, question, dan feedback, yang mana feedback akan dikelola oleh admin (superuser) yang terdaftar pada tbuser sebagai is_superuser. Seluruh relasi yang berkesinambungan proses bertujuan untuk memastikan bahwa data saling terhubung dan dapat digunakan untuk berbagai tujuan, terutama dalam pengelolaan fungsi chatbot.
[0183] (Heading 2) 5.1.2.5.2 Vektor Database
[0184] (Normal) Vektor database merupakan basis data yang menyimpan data hasil embedding ke dalam vektor berdimensi tinggi. Vektor database sebagian besar digunakan dalam kepentingan data yang tidak terstruktur seperti gambar, teks, atau suara untuk keperluan AI ataupun similarity search. Vektor database dijadikan sebagai database dinamis yang menyimpan informasi tanpa perlu membuat kolom yang kaku pada awal proses seperti RDMS. walau demikian, vektor database dalam penggunaan akan tetap dibatasi dalam beberapa kolom utama yang digunakan sistem untuk proses sistem knowledge-based (RAG). Gambar 5.29 mengilustrasikan database vektor yang dirancang dalam qdrant database.

[0457] (Normal) Cross attention dapat ditemukan pada tahap decoder yang menggunakan lapisan encoder. [0598] (Normal) Implementasi alur kerja RAG memberikan kemampuan chatbot untuk menjawab pertanyaan berdasarkan sumber pengetahuan, yaitu dokumen PDF.
[0599] (Normal) Implementasi RAG menggunakan library fitz untuk ekstraksi dokumen, SentenceTransformers untuk embedding, Qdrant untuk menyimpan data vektor, hingga model Large Language Model (LLM) berupa SeaLLMs untuk generate hasil akhir.
[0600] (Normal) Alur kerja sistem dimulai ketika sistem melakukan pemotongan teks (Chunk) pada setiap paragraf pada dokumen pdf yang diinput oleh pengguna. Sistem memotong setiap paragraf menjadi beberapa kalimat yang nantinya akan diubah menjadi vektor. Proses akan diawali dengan ekstraksi teks dengan fungsi extract_pdf_text yang disediakan oleh library fitz. Sistem juga disertai dengan deteksi kolom berdasarkan sumbu X untuk mencegah adanya kolom teks. Teks yang berhasil diekstrak dari proses sebelumnya kemudian akan dilakukan tokenisasi oleh NLTK dan dilanjutkan dengan proses_chunk yang mengelompokkan kalimat menggunakan teknik sliding window dengan parameter ukuranchunk=5 dan kalimatchunk=2. Parameter chunk diberikan dengan memastikan bahwa tidak terjadi tumpang tindih pada kalimat sehingga tidak ada informasi pada teks yang terhapus atau terpotong.
[0601] (Normal) Hasil dari proses chunk kemudian akan memanggil model paraphrasemultilingual-MiniLM-L12-v2 sehingga model mengubah chunk menjadi vektor 385 dimensi dan kemudian disimpan pada Qdrant dengan fungsi yang telah dibuat, simpan_ke_qdrant. Skenario testing pada signup yang dibangun berupa: permintaan GET sehingga merender halaman signup.html, permintaan POST dengan data yang sudah ada di database (mengembalikan error JSON), permintaan POST dengan data valid (mengembalikan success JSON dan menyimpan pengguna baru).
